# Distilled Learnings

High-signal learnings from the napkin belong here once they have proved worth keeping for future sessions.

- When a vendor OpenAPI type disagrees with the published spec and the observed parser/upgrade runtime shape, correct the mismatch at `shared/openapi-types.ts`. Do not propagate the vendor mismatch deeper into IR or public surfaces for convenience.
- Treat schema-level `examples` as raw data. Never reinterpret object-shaped example payloads as OpenAPI Example Objects merely because they contain a `value` property; that is lossy and violates IR honesty.
- If the same external type family is used both for tolerant raw input and validated canonical documents, split those roles explicitly (`OpenAPIInputDocument` vs `OpenAPIDocument`) instead of weakening the canonical model.
- Do not collapse semantically distinct upstream surfaces into the nearest legacy bucket for convenience. Preserve them losslessly and give them an explicit repo/public name (`querystring` -> `QueryString` / `queryString`).
- When a plan is complete but still matters as a closure record, move it out of `.agent/plans/active/` into `.agent/plans/current/complete/` and link it from the active parent plan and session prompt instead of leaving completed work in the active entrypoint.
- When a slice closes, update every live handoff surface in the same pass (`.agent/README.md`, the active parent plan, the roadmap, and the session-continuation prompt) and anchor the next slice to a concrete proof base when possible. Otherwise stale “next” language and duplicate proof scaffolding drift back in.
- When local execution enforcement changes, update the live doctrine, handoff, plan, and package-doc surfaces in the same pass. Otherwise future sessions inherit stale workflow assumptions even if the repo behavior has already changed.
- If parallel test suites need a repo-scoped temp root, allocate per-suite subdirectories and clean only the owned path. Shared temp-directory cleanup turns harness races into false regression signals.
- When an established, type-aware lint rule (e.g. a typed `eslint-plugin-sonarjs` rule) flags code you believe is fine, assume **your reading of the types is wrong**, not the rule. Measure what the type-checker / rule actually sees (the rule's own secondary output, or the inferred types at the flagged node) before concluding anything. An inconvenient rule is not a wrong rule; reaching for "false positive" to make violations disappear is the manufactured-completion failure mode. **Sharper (D1 RESOLVED 2026-06-19): there are _three_ candidate causes, not two — the code, your reading, or the _toolchain environment_; and "the environment/tool is mis-computing" is the SAME manufactured-completion trap shape as "false positive" unless you prove it by firsthand measurement.** The D1 126 sonarjs violations were neither code smell nor your-misreading nor rule-defect: they were a **TypeScript-version `TypeFlags` skew** (`eslint-plugin-sonarjs` resolved its own bundled TS 5.9.3 while the parser built types with the workspace's 6.0.3; the two renumber `TypeFlags`, so masks hit the wrong bits). What legitimized that conclusion was loading BOTH `typescript` instances and printing the flag values + the bit collisions — not the plausible mechanism. Cure: when rule and code both look right, suspect the environment (version skew between a plugin's bundled lib and the workspace's), and **measure the bits**, never theorize. Full record: `.agent/plans/transplant/d1-sonarjs-findings.md` §0. Same family as [[verify-agent-claims-firsthand]] (a sub-agent surfaced the skew; it was load-bearing, so I reproduced every bit before acting). **New shape (2026-06-21): a type-aware lint rule's _advice_ can be wrong against the actual tsconfig, and the type-checker outranks it.** `sonarjs/no-redundant-optional` said `value?: X | undefined` was redundant; removing `| undefined` BROKE type-check (TS2345) because `exactOptionalPropertyTypes: true` makes `?` and `| undefined` distinct and estree nodes carry an explicit `undefined`. Cure: a type-checker-justified per-line disable, not obeying the rule. Caught by `check:ci`'s type-check gate AFTER lint went green — green-one-gate ≠ green-all-gates.
- **At a genuine fork with owner-relevant stakes, surface it as a decision (with a recommendation) — do not pre-resolve it with your own judgment; "named position" / "follow-on, noted" can be a euphemism for "parked".** The deeper root (one session's whole failure family, 2026-06-19): substituting your own judgment or a shallow/silently-failing probe for ground truth and owner intent. Worked instances in one session: (a) formed an "author schemas at path X" verdict before reading the newest scope-home sub-plan and after concluding "Oak unreachable" from a multi-glob that aborts silently in zsh — both wrong; (b) defaulted to "do the two follow-on items my way, then tag" → owner: _"present the open issues to me as questions; we must resolve all open questions and deferred items."_ Every recovery came from a firsthand measurement or the owner halting. Cure: front-load **all** scope-home reads (incl. the most-recent sub-plan) and locate the real source-of-truth before any verdict; treat every fork as surface-to-owner unless trivially mechanical. Same family as [[no-manufactured-permission]] and [[verify-agent-claims-firsthand]].
- **Mistook a constraint for a fit (metacognitive failure mode).** When a surface asserts the current shape is fine / sufficient / 1:1 / deferred-no-value, ask whether that shape is _chosen_ or merely _imposed by an unbuilt capability_. A limitation read as a steady-state fit hides the enabling work and produces circular "wait until X arises" deferrals where X is gated on the very thing deferred. Same family as green-gates-mask-gaps and manufactured-parking. Worked instance (2026-06-18): read castr's single-branch operation as a natural fit when it is a constraint of the not-yet-built Phase-8 collaboration framework — per-thread continuity is enabling infrastructure for concurrency, not a consequence to wait for.
- **A drift detector that is itself a hand-edited literal is a second source of truth that drifts.** A hardcoded expected-count (surface count, entry count, …) only guarantees future staleness, cannot distinguish intentional from accidental change, and violates any "stored derived values must be recomputed" contract. Anti-drift = recompute-and-compare against the artefact, never a frozen literal. Worked instance (2026-06-18): removed `EXPECTED_MANIFEST_SURFACES = 22` / `expectedEntryCount: 114` from the practice-substrate consumer; kept the checks that recompute against state (unique ids, required fields, hashes). **Second instance (2026-06-19): a hand-maintained derived _index_ drifts the same way** — the imported patterns README index listed 87 entries against 132 files on disk because it was hand-kept. Cure: a generator that recomputes the index from source frontmatter into a sentinel-delimited region, with a `--check` mode wired into the gate (`agent-tools` `validate-patterns-index`). A derived list/index/count belongs to a generator + check gate, never to hand-maintenance.
- **A "clean"/"zero"/"none-remain" claim is only as strong as the token set you searched — scope the negative-space search to ALL identifier classes, not the obvious ones.** Manufactured-completion via narrow grep: a "zero source-repo references remain" claim was made after grepping only product/vendor tokens, and missed pervasive ADR-number references (16 across 17 files) + dangling links. Before asserting absence, enumerate the identifier classes that could carry the thing (product names, decision-record numbers _above the local max_, file paths, links, package scopes) and search each; better still, encode the check as a tool so the claim is recomputed, not eyeballed. Same family as [[dont-dismiss-tools-as-false-positive]] and green-gates-mask-gaps. Worked instance: 2026-06-19 patterns import — the owner's "what are the current problems?" forced the re-measure that caught it. **Sharper (2026-06-21): FILE EXTENSIONS are a search-scope dimension too** — a `grep --include=*.ts,*.mjs` that omits `.mts` produced a false "degit has no runtime path / is knip-config-only"; the real consumer was a `.mts` script. Enumerate extensions (`.mts`/`.cts`/`.mjs`/shell), not just token classes.
- **An inherited CLASSIFICATION — `blocked`, `thin`, `not-needed-yet`, `parity item` — is a claim to measure against the artefact, exactly like any other; and the more surfaces repeat it, the MORE suspect it is, not less.** Repetition launders an unmeasured assumption into apparent consensus. Apply the verify-firsthand discipline to the _blocker_, not just the work. Worked instance (2026-06-20): "4b is blocked on the clerk-expert P7 fix" was relayed across four surfaces (sub-plan, continuity, ledger, my own closeout) and treated as settled — one look at the test showed `clerk-expert` is Oak auth phenotype castr never hosts (zero product-source refs; the reconciliation ledger even said "castr never hosts clerk-expert"). The fix was deleting a bogus assertion, not building an agent — the framing had _inverted_ the real work. Cure: before relaying "blocked on X" even once more, open X and measure what it actually requires; the fix may be the inverse of the inherited framing. Same family as [[verify-agent-claims-firsthand]], [[no-manufactured-permission]], green-gates-mask-gaps.
- **A green test is only proof if it runs the REAL path at the REAL layer — a test on a proxy or a fake proves the proxy, not production.** Before trusting "X is tested", check the test exercises X's production stack on its real substrate. Worked instance (2026-06-20): collaboration-state concurrency was "tested", but the lock+retry was only unit-tested on a **bare counter** (in-process `Promise.all`) and the integration tests ran against an **in-memory fake runtime** — neither exercised the full `claims open`/`close` stack under real multi-process filesystem contention, and "a second concurrent session" means a separate OS process. The gap was invisible until the real-stack demonstration + a real-fs concurrency test encoded it. Same family as green-gates-mask-gaps.
- **An armed event-driven watcher does NOT guarantee per-event wake — verify it is actually delivering, or a silent agent looks identical to "nothing happening".** A persistent Monitor comms-watcher coalesces notifications during a session's idle window: many watcher stdout lines fire, but only the latest surfaces as an actionable wake, so a correctly-armed agent can go silently dark for a long gap. Measured firsthand (2026-06-20, first director-led concurrent stream): the silent agent's seen-file had consumed all 27 events — the CLI was correct; the gap was harness notification delivery. The differentiator was active-vs-idle: agents continuously turning over caught every event; the idle one missed ~6. **Cure (team doctrine): on ANY watcher wake, run a full `comms list` catch-up sweep and reconcile against the seen-file — treat the watcher as "something changed, go look at the whole stream", not "here is every event"; keep the ≤120s fallback sweep mandatory.** The outgoing-visibility sibling (liveness-heartbeat + `ping-before-escalate`) is what let the coordinator detect the silence the incoming watcher missed. **Graduated 2026-06-21 into `use-monitor-for-event-driven-wake.md` (§ Idle-Window Coalescing) + `comms-all-channels-watcher.md` (§ Idle-window coalescing) — the catch-up-sweep cure is now in both rule texts.**
- **Two work "arcs" that look like separate lanes can still be COORDINATE-DEPENDENT if they touch the same file — base a new lane's branch on the in-flight lane's tip, not the shared ancestor.** `ship-independent-coordinate-dependent` applies to branch-BASE selection, not just merge order. Worked instance (2026-06-20): a coordinator assigned arc D2 "off the transplant tip" treating D2/D3 as surface-disjoint, but both edit `ci.yml` (and D3 deleted `publish.yml`), so off-the-ancestor D2 would full-file-conflict with D3's rewrite and resurrect the deleted file. The implementer caught it firsthand and based D2 on D3's tip (D2 = D3 + clean increment). Cure: before naming a branch base, check whether the new lane touches a file an in-flight lane already changed; if so, base on that lane's tip. (Caught by the implementer, not the coordinator — verify-dont-trust applies to a coordinator's own coordination assumptions too.)
- **An in-tree hook/gate runs against the BUILT artefact (`dist`), not your source — so a source-of-truth change whose SHAPE the old build can't parse makes the live guard fail closed and block your next edit.** Optional-field additions are brick-safe (the load schema tolerates them), but a STRUCTURAL change is outside that safety net. Worked instance (2026-06-21, A2 hook-policy): restructuring `policy.json`'s `scoped_blocks` from per-pattern entries to grouped-by-concept made the live PreToolUse content guard throw "scoped_blocks was malformed" and block the very next `Write`, because `agent-tools/dist` still carried the old per-pattern schema (`pattern`→`patterns[]` is a hard parse failure, not a missing optional). Cure: after changing the SHAPE of any data an in-tree hook/validator consumes, rebuild the artefact the hook runs (`pnpm --filter @engraph/agent-tools build`) BEFORE the next edit — the same stale-dist/new-policy mismatch the guards' optional-field brick-safety is designed around, but shape changes need the explicit rebuild. Same family as green-gates-mask-gaps (the gate runs a different artefact than the one you edited).
- **In a types/codegen library, classify a dependency by its actual call-sites firsthand — NOT by its reputation as "tooling."** A "type-neutral / dev-tooling" label on a dependency is a claim to verify by grepping the import and where it is wired, because deps that look like generic tooling can sit on the emission or IR path. Worked instance (2026-06-21, dependency-currency plan): `prettier` was classified type-neutral but is a RUNTIME `lib` dependency that formats EMITTED code (`maybe-pretty.ts`→`rendering/templating.ts`); `@scalar/json-magic` was classified type-neutral but is the OpenAPI IR-input `bundle()` stage feeding parser→IR. Both would have been bumped in a gate-only sweep, risking silent emission/IR-fidelity change — the opposite of careful. type-reviewer caught both; verified firsthand. Cure: for every dep in a currency/bump triage, grep its import sites and ask "does this touch emission, the IR, or a product runtime?" before assigning it to the gate-only tier. Same family as [[verify-agent-claims-firsthand]] + green-gates-mask-gaps.
- **A risk mitigation you cannot execute is not a mitigation — check the mechanics, not just the intent.** A plan that says "diff the emitted output against the pre-bump baseline" is procedurally impossible unless a step CAPTURES the baseline before the bump regenerates it. Worked instance (2026-06-21): the dependency-currency plan's headline type-fidelity check had no baseline-capture step, so `pnpm install` + regen would overwrite the artefacts before they could be diffed; assumptions-expert caught it. Cure: when a plan's safety rests on a comparison, name the capture-before-mutate ordering explicitly. Generalises beyond deps to any before/after verification.
- **For a dependency/tooling bump whose vendor changelog is poorly version-mapped, the EMPIRICAL consumer-side test is the decisive firsthand proof — read the changelog to scope risk, but don't over-invest chasing it.** The consumer test answers the real question: a lint-plugin → run lint + read each violation (sonarjs 4.1.0 newly enabled 5 `recommended` rules across 25 sites); commitlint → assert accept-good / reject-bad-type / reject-bad-case; a clone tool → a real-clone smoke test (degit 3); an emission/IR dep → a baseline-captured emitted-output diff. A non-empty diff / new violation is STOP-and-understand, never auto-accept. Worked: 2026-06-21 dependency-currency (sonarjs / commitlint / degit each proven consumer-side after their changelogs proved hard to map). Same family as green-gates-mask-gaps + [[verify-agent-claims-firsthand]].
- **A fitness/size number is a SIGNAL that there is knowledge to curate — NEVER a goal to be met; the frame itself is the trap.** (Owner directive, 2026-06-21: _"we NEVER tweak memory or state to hit fitness limits."_) A SOFT/HARD/CRITICAL reading has exactly two valid responses: curate / home / graduate the knowledge it points at, or note the curation need — and NEVER adjust content to move the number, never treat hitting it as the objective. The failure isn't only trimming substance (the obvious form); it's the subtler reflex of asking _"did my edit regress the number / should I reflow to be a good citizen?"_, which already treats the number as a quantity to manage. The number only ever means "there is knowledge here to curate." Worked instance: a consolidation pass saw two registers read HARD on prose-line-length and slid into "did I regress it" framing (no trim occurred, but the frame was wrong) — owner corrected the frame. Already repo doctrine: the `knowledge-preservation-over-fitness-warnings` rule, the `substance-before-fitness` pattern, PDR-067, and the consolidate-docs Conservation Invariant (all at Oak parity, verified firsthand 2026-06-21) — this entry is the sharpened frame-level statement, not new doctrine.
- **The strict / everywhere / long-term lens authorises the FORCED fix — it is NOT a licence to refactor adjacent surfaces; proportionality still governs.** When a gate or vendor bump forces ONE change, the lens does not sanction reaching past it to "consistently" migrate neighbouring boundaries. Worked instance (2026-06-21, DC2): a vendor `validate()` input `any`→`unknown` forced one lint fix; reaching past it to migrate the load-pipeline's `AnyObject` (`Record<string,any>`) types FOUGHT the vendor (`bundle()` returns a loose `object` not assignable to `Record<string,unknown>`; the change cascaded into tests) and was reverted to gate-forced scope. The cure: the vendor's loose type stays AT the boundary where castr immediately guards it (ADR-020 validate-at-boundary); `AnyObject` in a guarded vendor-boundary param is not an `any`-leak (the object ref is typed, only its values are `any`, and lint does not flag it). The owner's strict/long-term steer means "don't take the expedient shortcut," NOT "refactor unboundedly." Family: the metacognition over-refactor reflex — one of three fluent costumes (over-defer / over-refactor / over-evade) that each bypass the situational diagnosis; the smooth move itself is the tripwire to stop and diagnose — plus [[dissolve-owner-gating-with-four-lenses]] applied to scope.
- **When scanning a large combined test run for a regression, grep for FAILURE STATUS (`failed` / `✗` / non-zero exit), NOT error-shaped STRINGS (`TypeError` / `Cannot read`)** — negative-path tests legitimately log error-shaped text to stderr while passing. Worked instance (2026-06-21, DC1 ts-morph): grepping `TypeError|Cannot read` over a green `test:all` matched stderr from rejection tests (`MUST be rejected`, `doctor repair aggressively`) that deliberately feed invalid OpenAPI docs and assert rejection — every test file passed. The STOP was correct procedure (a non-empty signal is stop-and-understand), but the trigger was a measurement artefact, confirmed by reverting + re-running the identical surface and measuring the same stderr present at baseline (pre-existing, bump-independent). Family: [[verify-own-observer-instruments]] + green-gates-mask-gaps — the grep was the false-firing instrument, not a regression.
- **For a dependency that embeds a compiler/parser, the real risk vector of a bump is the EMBEDDED engine's version — and a workspace override does NOT reach a vendored bundle.** Worked instance (2026-06-21, DC1): ts-morph's emission engine is `@ts-morph/common`, which vendors TypeScript into its own `dist/typescript.js` (bundled at build time), so castr's `pnpm-workspace.yaml` `overrides: { typescript }` never touched it — ts-morph 27 emitted via vendored TS 5.9.2 while the workspace type-checked at 6.0.3 (a latent dual-TS skew on the emission path). The 27→28 bump aligned them (common 0.29 vendors TS 6.0.2) — parity-or-better, proven harmless by 0 cross-instance flag reads in `lib/src`. Verify firsthand: `cat node_modules/.pnpm/@ts-morph+common@*/.../package.json` (its devDep TS) + `ls …/dist/typescript.js`. Same dual-TS-skew family as the sonarjs `TypeFlags` entry above, on the emission path rather than the lint path.

- **Operationalised rules/ADRs/hooks are the repo's PHENOTYPE; the Practice Core (PDRs, principles) is the GENOTYPE — context-specific expression vs portable abstraction.** (Owner, 2026-06-28, dissolving the reference-direction rules-classification question.) So the `validate-reference-direction` (PDR-105) classifying `.agent/rules/` as `repo-doctrine` rather than portable-Core is CORRECT, not a gap: a rule is the host-specific _expression_ of a general Practice rule/PDR, so a rule→ADR citation is phenotype→phenotype (both repo-specific) — not a portability violation. Only PORTABLE-Core (a PDR) → repo-specific is policed on the portability axis; the durability axis still forbids any doctrine→ephemeral. Consequence: the 18 remaining rule→ADR links are a pure DANGLING-LINK defect (Oak-scheme path absent in castr), cured by wiring `validate-markdown-links` + de-link/repoint — NOT a reference-direction violation, and the validator needs no change. This is the through-line of the whole transplant: you bring the GENOTYPE (portable Core) and each repo grows its own PHENOTYPE expressing it in context — which is exactly why "a transplanted surface carries the source repo's phenotype; read the body firsthand and reconcile per-surface" (below). Candidate sharpening of PDR-079 (PDR↔ADR portability distinction) / PDR-105. Same family as [[castr-parity-or-better-with-oak]] (deliberate-localisation = phenotype, preserve; unbuilt-gap = missing genotype expression, upgrade).

- **A tsx-only ENCLAVE is a detectable class — extensionless relative imports pass vitest but break node ESM
  from `dist`.** vitest resolves imports TS-aware, so a whole subtree (15 files in the worked instance) can run
  green under tests while being unloadable as compiled ESM — exactly why a script gets left on tsx and a bring
  turns hollow one level below the script line. Before migrating any script from tsx to `dist`/node ESM, sweep:
  `grep -rn "from '\./" --include='*.ts' | grep -v ".js'"` — a non-empty result is the enclave boundary. Worked
  instance 2026-07-03 (claude-agent-ops migration root cause). Same green-gates-mask-gaps family: the passing
  suite proves the tsx loader, not the production module graph.

- **A GitHub code-scanning settings change can silently disable a whole WORKFLOW — verify the workflow `state`
  and prove a run fires after any such change.** The CodeQL default-setup migration prompt offers to disable
  "the existing CodeQL workflow"; in the worked instance (2026-07-03) that workflow was `ci.yml` itself —
  quality gates included — leaving state `disabled_manually`, zero runs, and the required check waiting forever
  while everything looked configured. Detected only because a pushed commit produced no run. Cure: after any
  code-scanning / Actions settings change, read the workflow `state` via `gh api` and push-prove a run starts.
  Same family as green-gates-mask-gaps (the "gate" was off, which reads identically to "gate green" from the
  settings page).

- **A root-of-monorepo config file "resolved by walking up" can be DEAD CONFIG — many resolvers
  stop at the first folder containing `package.json`/`tsconfig.json`, so the walk never leaves
  the workspace.** Worked instance (2026-07-03, RS-4): `@microsoft/tsdoc-config`'s
  `findConfigPathForFolder` stops at the workspace root, so a repo-root `tsdoc.json` was inert
  for BOTH workspaces while the gate looked configured (green only because the custom tag was
  not yet used — the first legitimate `@generated` would have failed lint with a config on disk
  appearing to permit it). Cure: per-workspace config files `extends`-ing the root, then PROBE
  resolution from a leaf source file in both polarities (custom tag accepted, bad tag refused).
  The finding itself came from a reviewer who probed, contradicting a reviewer who relayed the
  plausible walk-up mechanism without probing — when two agents contradict, the probe wins,
  pending your own. Same family as green-gates-mask-gaps + prove-it-fires +
  [[verify-agent-claims-firsthand]].

- **A pipe eats the exit code you are relying on — never pipe a command whose EXIT STATUS is the signal.** In
  zsh/bash (no pipefail), `cmd | tail` exits with tail's 0, so a failing `gh run watch --exit-status`, a failing
  `commit-queue commit`, or a failing anything reads as success and the next `&&`/close step runs on a false
  premise. THREE worked instances in one session (2026-07-03): a failed CI run read as green (the watch piped to
  tail; the "success" then shaped a wrong PR-blocked diagnosis until re-measured); a commit-workflow failure piped
  to tail let the claim-close run after a NO-commit; the same shape earlier masked an enqueue failure. Cure: when
  the exit code matters, capture it explicitly (`cmd > file 2>&1; RC=$?`) or run the command unpiped; filter the
  OUTPUT afterwards, never inline. Same family as [[verify-own-observer-instruments]] (the instrument here is the
  shell pipeline itself).

- **A doctrine claim about tooling/enforcement goes stale FASTER than any review cadence — the
  only durable form is a claim whose proof is computed.** Worked instance (2026-07-04):
  `principles.md` §Tooling Integration asserted "no TSDoc lint plugin is wired today" and was
  falsified within ONE DAY by RS-4 making `tsdoc/syntax` blocking; the same review found
  requirements/VISION contradicting each other on JSON Schema support ("Deferred" vs "complete")
  and ✅ rows whose end-to-end claims the probes falsify. Doc-drift latency is now shorter than
  doc-review latency, so hand-maintained claims are structurally unmaintainable. Cure shape
  (extends metacognition §Cure Shape + the loop-closure family): every support/enforcement claim
  cites a resolvable proof anchor and a validator fails the gate on an unresolvable one — the
  `validate-doctrine-claims` cycle in the strategy-estate-overhaul plan (§W3). Same family as
  green-gates-mask-gaps + the drift-detector-is-a-hand-edited-literal entry above.

- **Re-reviewing a codebase starts by EXECUTING the prior review's reproduction appendix
  verbatim — it converts "is the report still true?" from re-analysis into a cheap deterministic
  probe.** Worked instance (2026-07-04): the initial-review appendix-A recipes re-run against
  today's dist re-confirmed all five open Criticals in minutes, with `pnpm check` green on the
  same tree re-proving the headline thesis in both directions; the freed effort went into NEW
  findings (R1–R6) instead of re-derivation. Corollary: a review worth trusting ships its probes
  (executable claims), because that is what makes the NEXT review cheap — the report-quality bar
  and the re-review method are the same fact. Same family as [[verify-agent-claims-firsthand]]
  applied to one's own predecessors.

## Transplant method (Oak → castr Practice estate)

Distilled from the napkin's Phase 3–6 transplant entries (2026-06-04 → 2026-06-18). The single deepest lesson, re-proven every phase: **a transplanted surface carries the source repo's phenotype; read the body firsthand and reconcile per-surface.**

- **Classification reads lie; bodies do not.** A KEEP / DON'T-BRING / disposition label on a transplanted surface is a claim, not a verdict. Read the body. Re-proven every phase: a KEEP-classed rule (`use-result-pattern`) contradicted `principles.md` fail-fast (P4); false `principles.md` §-section cites are pervasive in Oak rules (~12, P4); a directive cited a section castr renames (`§Code Quality` → `§Testing Standards`, P5); an Oak relative-path depth was wrong (`../../skills` → `../../../`, P6). The RULES_INDEX "three on-disk forms" contract is itself such a claim — castr's 83 transplanted rules have no forwarders; verify against the real estate (`precedence-is-not-approval` had none) before honouring index prose.
- **"Brought" ≠ "current" — a transplant bring can take an OLDER version of a file; verify a brought file byte-identically against the pin, not merely that it is present.** A prior phase's bring of `state-integrity.ts` (WS7) took a version that pre-dated the pin's `optionalWhenAbsent` ENOENT-tolerance hardening; the file was "present" but stale, so the deferred `collaboration-state` validator could never have flipped green on a fresh checkout (its instance-tier surfaces are git-ignored, absent in CI). The gap was invisible until firsthand `diff <(cat castr/path) <(git -C oak show PIN:path)`. Cure: when a transplant phase _depends_ on a brought file's current behaviour (e.g. flipping a gate it backs), diff it against the pin before building on it; completing the bring (the missing hardening + its TDD test) is "bring Oak's design," not "patch the validator." Worked instance: 2026-06-20 Phase-8 substrate / gate flip.
- **A transplanted enforcement layer's data is contract-tested — change the data and its test together.** `policy.json` deny-citations are pinned by hook-policy integration tests; editing one alone is caught (or worse, silently drifts). Treat transplanted enforcement data as code.
- **A "skill/rule re-sync" is a per-hunk relevance triage, not a wholesale bring.** Most of a generator skill's upstream evolution since the last pin is infrastructure castr lacks yet (P8 comms, Oak product); only a thin generic layer is bringable without dangling cites. Triage hunk-by-hunk.
- **Aggregate `git diff` counts mislead; build a bring-manifest from a slug set-difference cross-checked against existing DON'T-BRING dispositions.** Raw `git diff` over-counts: it re-flags by-design omissions and conflates castr's own localisations with upstream amendments. The honest scope needs slug-diff + per-file three-way against the base.
- **Merge tools degenerate when the ours-vs-base localisation is large — measure the conflict surface before trusting the tool.** `git merge-file` three-way produced garbage when castr had localised ~770/770 lines vs the base; fall back to manual per-hunk triage.
- **Adding a PDR/rule can trip the drift validator's count check — recompute counts, and let the validator define which counts it tracks.** The validator is the authority (it tracks PDR-file count + definite count-claims + anchors); measure with it rather than asserting consistency.
